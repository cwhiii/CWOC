# Requirements Document

## Introduction

This feature enables birthday and anniversary dates stored on contacts to automatically generate recurring annual calendar entries. Each date on a contact has a per-date toggle (defaulting to enabled) that controls whether it appears on the calendar. The calendar search functionality correctly finds these birthday/anniversary entries across multiple years.

## Glossary

- **Calendar_Engine**: The frontend calendar rendering system that displays chits and virtual events on day/week/month/year views.
- **Birthday_API**: The backend endpoint (`/api/contacts/birthdays`) that generates virtual calendar entries from contact date fields.
- **Contact_Editor**: The frontend page for creating and editing contacts, including their date entries.
- **Date_Entry**: A single date record on a contact, consisting of a label (e.g. "Birthday", "Anniversary"), a value (YYYY-MM-DD), and a `show_on_calendar` flag.
- **Virtual_Calendar_Entry**: A chit-like object generated by the Birthday_API that represents an annual occurrence of a contact date, displayed on the calendar but not stored as a standalone chit.
- **Search_System**: The frontend and backend search functionality that finds chits and virtual calendar entries by text query.

## Requirements

### Requirement 1: Annual Calendar Entry Generation

**User Story:** As a user, I want birthday and anniversary dates on my contacts to automatically appear as annual events on my calendar, so that I never forget important dates.

#### Acceptance Criteria

1. WHEN the Calendar_Engine requests events for a date range, THE Birthday_API SHALL return Virtual_Calendar_Entries for each contact Date_Entry that has `show_on_calendar` set to true.
2. THE Birthday_API SHALL generate Virtual_Calendar_Entries for the previous year, current year, and next year relative to today, so that calendar navigation across year boundaries displays entries correctly.
3. WHEN a contact Date_Entry has an original year earlier than the current year and later than 1900, THE Birthday_API SHALL include the age in years in the Virtual_Calendar_Entry title.
4. THE Birthday_API SHALL format each Virtual_Calendar_Entry title as "🎂 {display_name} — {label} ({age} yrs)" when a label and valid original year exist.
5. THE Birthday_API SHALL set each Virtual_Calendar_Entry as an all-day event spanning 00:00:00 to 23:59:59 on the anniversary date.
6. WHEN a contact Date_Entry falls on February 29 and the target year is not a leap year, THE Birthday_API SHALL generate the Virtual_Calendar_Entry on February 28 of that year.
7. THE Birthday_API SHALL use the contact's color for the Virtual_Calendar_Entry, falling back to "#f5e6d3" when no contact color is set.
8. THE Birthday_API SHALL include the contact's ID, image URL, and date label as metadata fields on each Virtual_Calendar_Entry so the calendar can link back to the contact.

### Requirement 2: Per-Date Calendar Toggle

**User Story:** As a user, I want a checkbox on each date entry in the contact editor that controls whether that date appears on the calendar, so that I can selectively show only the dates I care about.

#### Acceptance Criteria

1. THE Contact_Editor SHALL display a "Show on Calendar" checkbox next to each Date_Entry input row.
2. WHEN a new Date_Entry is added, THE Contact_Editor SHALL set the "Show on Calendar" checkbox to checked by default.
3. WHEN the user unchecks the "Show on Calendar" checkbox and saves the contact, THE Birthday_API SHALL exclude that Date_Entry from Virtual_Calendar_Entry generation.
4. WHEN the user re-checks the "Show on Calendar" checkbox and saves the contact, THE Birthday_API SHALL include that Date_Entry in Virtual_Calendar_Entry generation.
5. THE Contact_Editor SHALL persist the `show_on_calendar` boolean as part of the Date_Entry JSON object stored in the contact's `dates` field.

### Requirement 3: Calendar Display Integration

**User Story:** As a user, I want birthday and anniversary entries to appear seamlessly alongside my regular calendar events with a distinctive shape, so that I can instantly recognize them among my schedule.

#### Acceptance Criteria

1. WHEN the Calendar_Engine renders a day, week, month, or year view, THE Calendar_Engine SHALL fetch Virtual_Calendar_Entries from the Birthday_API and display them alongside regular chits.
2. THE Calendar_Engine SHALL visually distinguish Virtual_Calendar_Entries from regular chits by displaying the 🎂 emoji prefix in the title.
3. THE Calendar_Engine SHALL render Virtual_Calendar_Entries with a distinctive "concave notch" shape: a rectangle with concave (inward-pointing) triangular cutouts on the left and right edges, visually the inverse of the pointed/arrow shape used for point-in-time chits.
4. THE Calendar_Engine SHALL pad the text content of Virtual_Calendar_Entries inward so that text does not extend into the concave triangular areas on either end.
5. WHEN a user clicks a Virtual_Calendar_Entry on the calendar, THE Calendar_Engine SHALL navigate to the associated contact in the Contact_Editor.
6. THE Calendar_Engine SHALL display the contact's profile image on the Virtual_Calendar_Entry when an image URL is available.

### Requirement 4: Search Across Years

**User Story:** As a user, I want to search for birthday and anniversary entries and find them regardless of which year they occur in, so that I can quickly locate upcoming celebrations.

#### Acceptance Criteria

1. WHEN a user searches for a contact name, THE Search_System SHALL include matching Virtual_Calendar_Entries in the search results.
2. WHEN a user searches for a date label (e.g. "Birthday", "Anniversary"), THE Search_System SHALL return Virtual_Calendar_Entries whose labels match the query.
3. THE Search_System SHALL match Virtual_Calendar_Entries across all generated years (previous, current, and next year) so that results are not limited to a single year.
4. WHEN the calendar is filtered by search text, THE Calendar_Engine SHALL filter Virtual_Calendar_Entries using the same text-matching logic applied to regular chits.

### Requirement 5: Data Integrity and Ownership

**User Story:** As a user, I want birthday calendar entries to respect contact ownership and vault sharing rules, so that I only see entries for contacts I have access to.

#### Acceptance Criteria

1. THE Birthday_API SHALL return Virtual_Calendar_Entries only for contacts owned by the authenticated user or contacts shared to the vault.
2. THE Birthday_API SHALL exclude Virtual_Calendar_Entries for soft-deleted contacts.
3. WHEN a contact is updated or deleted, THE Birthday_API SHALL reflect the change on the next calendar fetch without requiring manual intervention.
4. IF the Birthday_API encounters a Date_Entry with an unparseable date value, THEN THE Birthday_API SHALL skip that entry and continue processing remaining dates without returning an error.
